Spectre-BHI 漏洞复现分析

1 漏洞原理

1.1 攻击原理

要理解BHI的攻击原理，必须先回顾其前身Spectre-v2（分支目标注入，BTI）以及为其设计的硬件防御措施。

  1. Spectre-v2 (BTI) 的经典攻击模式
    • 目标组件：分支目标缓冲区 (Branch Target Buffer, BTB)。BTB是CPU内部的一个高速缓存，用于记录间接分支指令（如 jmp rax）曾经跳转过的目标地址 。
    • 攻击手法：攻击者在用户态（低权限）反复执行一个间接分支，使其源地址和恶意目标地址在BTB中形成一条映射记录。当内核（高权限）执行到相同源地址的间接分支时，CPU的分支预测器会错误地从BTB中命中攻击者“投毒”的记录，并推测性地跳转到攻击者指定的恶意代码片段（gadget），从而导致信息泄露 。
  2. 硬件防御的“失算”：eIBRS 和 CSV2
    • 防御逻辑：为了应对BTI，英特尔和Arm分别推出了eIBRS (Enhanced Indirect Branch Restricted Speculation) 和 CSV2等硬件缓解措施 。这些措施的核心思想是隔离不同权限域的分支目标状态。当CPU从用户态切换到内核态时，内核的间接分支预测器将不再使用用户态留下的BTB记录 。
    • 致命疏漏：当时业界普遍认为，只要隔离了分支目标，就能有效阻止跨权限的BTI攻击。然而，研究人员发现，这些硬件防御虽然成功隔离了分支目标，却没有隔离分支预测所依赖的另一个关键因素——分支历史。
  3. BHI的核心原理：利用共享的分支历史 现代CPU为了提高预测准确率，其间接分支预测器不仅依赖于分支指令的地址，还会参考一个全局共享的**分支历史缓冲区 (Branch History Buffer, BHB)**。BHB记录了最近执行的一系列分支的结果（如跳转或不跳转），形成了一个代表程序控制流路径的“模式”。BHI的攻击原理正是利用了这个未被隔离的共享资源 ：
    • 攻击者在用户态通过执行精心构造的一系列分支指令，向全局共享的BHB中“投毒”，写入一个特定的比特模式 。
    • 当CPU切换到内核态并执行一个间接分支时，分支预测器会结合该分支的地址和从BHB中读取到的、已被污染的分支历史来进行预测。
    • 由于分支历史已被攻击者操纵，预测器会被诱导产生错误的预测，使其推测性地跳转到一个并非由攻击者直接注入、但存在于内核地址空间中且对攻击者有用的“泄露小工具”（disclosure gadget）。

BHI的攻击范式从“告诉”CPU一个错误的跳转目标（BTI），转变为“欺骗”CPU的预测逻辑，让它自己根据被污染的历史“推导”出一个错误的跳转目标。

1.2 漏洞利用方式

什么是 eBPF

  • eBPF 是 “extended Berkeley Packet Filter”的缩写，可以把它理解为“运行在内核里的一个安全小型虚拟机”。你可以把一段受限的程序加载进内核，让它在特定的钩子（hook）上触发执行，比如网络包处理（XDP/tc）、系统调用/内核函数探针（kprobe/kretprobe）、tracepoint、cgroup、LSM 安全钩子、性能事件等。
  • eBPF 程序写起来通常是“受限 C 语言”，编译成 eBPF 字节码后通过 bpf() 系统调用加载到内核。加载时会经过“验证器（verifier）”严格检查，确保不会越界访问、不会死循环、不会破坏内核稳定性，因此能在内核中“安全”运行。
  • eBPF 提供“maps”（类似共享的数据结构，如数组、哈希表、per-cpu 存储、环形缓冲等）在内核与用户态之间交换数据，或者让多个 eBPF 程序共享状态。还有“helper 函数”让 eBPF 程序以受控方式调用内核功能（比如取时间、访问 map、获取包信息等）。
  • 常见用途包括：高性能网络处理（如 XDP 防护/负载均衡）、观测与故障诊断（低开销 tracing/profiling）、安全审计与加固（利用 LSM 钩子）、性能分析等。

什么是 JIT（就地编译）

  • JIT 是 “Just-In-Time compilation”的缩写，意思是“在加载/运行时把字节码编译成本机机器码”。相比解释执行，JIT 之后的代码运行速度更接近原生内核代码，性能高、时序更稳定。
  • 在 eBPF 里，JIT 发生在内核端：当 eBPF 程序通过 bpf() 加载后，内核把 eBPF 字节码转成当前 CPU 架构（如 x86-64、arm64）的机器指令，然后再在相应的 hook 上执行。

它们与 Spectre-BHI PoC 的关系

  • 这个 PoC 里，eBPF 充当“内核态 gadget”的载体：PoC 会加载特制的 eBPF 程序（被内核 JIT 成机器码），在特定的条件下（分支预测被污染，产生瞬态执行）让这些 JIT 代码在内核路径中短暂执行，从而把数据编码到缓存侧信道里。
  • eBPF 的 map 被用作攻击的“基础设施”：例如作为 Flush+Reload 的缓冲区、时间记录区，或者构造驱逐访问序列。这样用户态可以通过这些 map 观察到缓存命中/未命中，从而恢复瞬态泄露的数据。
  • 你在 main.c 里看到的“设置 eBPF 内存限制”（提高 RLIMIT_MEMLOCK）是为了让这些 eBPF maps 成功创建并锁驻内存，属于运行时环境准备，不是 Spectre-BHI 的技术点本身。JIT 的存在则保证 eBPF 在内核里以原生指令执行，带来足够的性能和精确的时序，从而更容易被分支预测/瞬态执行路径“命中”和观测到效果。

一句话总结

  • eBPF：在内核里安全执行小程序的机制，带有 map（数据结构）和 helper（受控能力），广泛用于网络、观测、安全。
  • JIT：把 eBPF 字节码动态编译成本机机器码，提高性能和时序确定性。
  1. 寻找或创造“泄露小工具” (Gadget) 攻击的第一步是在受害者（内核）的地址空间中找到一段可利用的代码。这个“小工具”必须具备一个特征：在推测执行期间，它会根据某个秘密值（如内核指针、密码的一部分）去访问一个特定的内存地址。
    • 初始利用方式：非特权eBPF 最初的PoC攻击巧妙地利用了Linux内核中的非特权eBPF功能 。eBPF允许普通用户向内核提交一段经过验证的字节码程序。这为攻击者提供了一个完美的“Gadget工厂”，可以在内核中精确地植入一个符合攻击需求的泄露小工具，极大地降低了漏洞利用的难度 。因此，早期的缓解措施就是建议禁用非特权eBPF 。
    • 进阶利用方式：“原生BHI” (Native BHI) 后续研究证明，即使禁用了eBPF，攻击者依然可以在内核庞大的代码库中找到天然存在的、可被利用的“原生gadget” 。研究人员开发了名为InSpectre Gadget的工具，在内核中发现了数千个潜在的gadget，并成功构建了不依赖eBPF的端到端攻击（CVE-2024-2201）。这证明了BHI是一个更根本性的威胁。
  2. 污染分支历史 (History Poisoning) 攻击者在用户态执行一段代码，其中包含一系列经过精确计算的条件分支。通过控制这些分支是否跳转，攻击者可以向全局BHB中写入一个特定的、可控的比特模式 。
  3. 触发内核执行与错误预测 攻击者通过系统调用（syscall）等方式，使CPU进入内核态，并执行一个包含间接分支的目标代码路径。当内核执行到这个间接分支时，其预测器会读取到被用户态代码污染过的BHB，从而被诱导跳转到第一步中找到的泄露gadget。
  4. 瞬态执行与侧信道攻击 CPU开始推测性地执行这个错误的gadget。Gadget会根据内核中的秘密数据访问特定的缓存行。例如，如果秘密字节是0x41，gadget可能会访问地址为 base_array + 0x41 * 4096 的内存。虽然这个错误的执行路径最终会被CPU发现并丢弃，但它在CPU缓存中留下的痕迹（某个缓存行被加载了）却被保留了下来 。
  5. 数据窃取 攻击者返回用户态后，立即通过经典的侧信道技术（如Flush+Reload）来探测缓存状态。通过测量访问不同缓存行的延迟，攻击者可以判断出哪个缓存行在瞬态执行期间被加载过，从而反推出那个秘密字节的值（0x41）。重复此过程，即可逐字节地窃取完整的内核敏感信息 。

综上所述，Spectre-BHI的原理和利用方式体现了微架构攻击的演进。它不再依赖于直接的状态注入，而是通过操纵预测逻辑的输入（分支历史）来间接实现攻击目标，成功地让为防御前代攻击而设计的复杂硬件机制形同虚设。

pocs/inter_mode/src/snippet.S

BHB 填充函数：

fill_bhb 的目的就是“往分支历史缓冲区（BHB）里写入一段你指定的分支方向序列”，从而污染/训练分支历史，使得后续内核态的间接分支预测（比如 do_syscall_64 中的间接调用）受到这段历史的影响，按你期望的方式发生错误预测。

  • %assign i 0 \n %rep MAX_HISTORY_SIZE … %endrep
    • 这是汇编器的宏展开：将接下来的小片段“完全展开 MAX_HISTORY_SIZE 次”（MAX_HISTORY_SIZE 在 targets.h 中定义，通常为 256）。这么做是为了避免用一个循环来产生分支结果（循环自身会引入额外分支），从而得到“固定布局、固定地址的一长串条件分支”。这种“完全展开”的分支序列更可控、可重复。
  • inc rdi
    • rdi 前移，指向下一个历史字节。
  • je next_%+i
    • 条件跳转：当字节为 1 时，分支“被认为 taken”；为 0 时，分支“被认为 not-taken”。
    • 关键点：跳转目标 next_%+i 就在这条指令的“下一条指令位置”，因此无论 taken 还是 not-taken，控制流都会落到同一个地方“next_%+i”。也就是说，功能行为完全相同，但分支预测器仍会记录一次“条件分支及其方向”。
  • next_%+i:
    • 本次分支的落点标签，占位用。

eBPF 寄存器&常用宏速查

  • R0：返回值寄存器；同时也常被当临时寄存器用
  • R1~R5：BPF 函数参数寄存器（BPF_REG_ARG1 等同 R1，…）
  • R6~R9：callee-saved，可当长期临时寄存器
  • R10：BPF_REG_FP，栈帧基址（BPF 栈大小 512B，向下生长）

常见宏（简化理解）：

  • BPF_MOV64_REG(dst, src)：dst = src（64 位移动）
  • BPF_MOV64_IMM(reg, imm)：reg = imm
  • BPF_ALU64_IMM(op, reg, imm)：reg op= imm（如 ADD）
  • BPF_ALU64_REG(op, dst, src)：dst op= src
  • BPF_LDX_MEM(size, dst, src, off)：从 [src+off] 读内存到 dst（size 可为 B,H,W,DW）
  • BPF_ST_MEM(size, dst, off, imm)：把立即数写到 [dst+off]
  • BPF_STX_MEM(size, dst, src, off)：把寄存器 src 写到 [dst+off]
  • BPF_LD_MAP_FD(reg, map_fd) / BPF_LD_IMM64_RAW_FULL(...)：把“映射句柄/伪指针”塞进寄存器，供 helper 使用或做解引用
  • BPF_EMIT_CALL(helper_id)：调用 BPF helper（如 ktime_get_ns、map_update_elem）
  • BPF_EXIT_INSN()：返回（R0 为返回值）
