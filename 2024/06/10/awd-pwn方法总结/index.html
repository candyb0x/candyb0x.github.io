pwn_waf

首先将其源码下载下来或者git clone下来到本地pwn_waf

需要修改的地方是makefile文件，我们要修改日志存放的位置，这边推荐的位置就是在pwn题目的目录下创建一个隐藏文件夹（以点.开头的文件夹名称），这样比较好找也比较方便，但是要保证有可读可写权限；如果没有，也可以选择默认的/tmp/.waf路径，/tmp一般都会具有这两个权限

注意这里的LOG_PATH路径设置有个坑，这里的路径设置跟xinetd配置文件设计的方式有关，配置文件位置/etc/xinetd.d/

service ctf
{
disable = no
socket_type = stream
protocol = tcp
wait = no
user = root
type = UNLISTED
port = 9999
bind = 0.0.0.0
server = /usr/sbin/chroot
server_args = –userspec=1000:1000 /home/ctf ./pwn
banner_fail = /etc/banner_fail
per_source = 10
rlimit_cpu = 20
}

如果配置文件与上述类似，这种就是通过/usr/sbin/chroot启动，将文件活动的根进行了一个更改，上述就是默认的初始路径为/home/ctf，且只能操作该目录下的文件（不要企图用../，这个玩意不行！不行！），所以在配置路径时，要以/home/ctf拼接进行配置，假设LOG_PATH=/home/ctf/.pwn_waf，则最后日志写入的文件位置为/home/ctf/home/ctf/.pwn_waf，然后又由于没有创建该文件，导致最后nc连接时回出现Open log [/tmp/.waf/13_57_07_36de8.log] file failed!错误

个人发疯：就是这个破问题，浪费了差不多三天的时间，最后突然悟了想着试试，顶不住了（希望不会有人也是因为这个问题浪费时间）

patch预备知识

  • 64位程序函数调用

在64位程序中的函数调用的过程中，前6个参数是通过寄存器去传输的，前六个参数依次存放在寄存器rdi、rsi、rdx、rcx、r8、r9中；前8个浮点数参数依次存放在浮点寄存器XMM0~XMM7；

  • 无符号跳转

    汇编指令 描述
    JA 无符号大于则跳转
    JNA 无符号不大于则跳转
    JAE 无符号大于等于则跳转（同JNB）
    JNAE 无符号不大于等于则跳转（同JB）
    JB 无符号小于则跳转
    JNB 无符号不小于则跳转
    JBE 无符号小于等于则跳转（同JNA）
    JBNE 无符号不小于等于则跳转（同JA）
    • 有符号跳转
    汇编指令 描述
    JG 有符号大于则跳转
    JNG 有符号不大于则跳转
    JGE 有符号大于等于则跳转（同JNL）
    JNGE 有符号不大于等于则跳转（同JL）
    JL 有符号小于则跳转
    JNL 有符号不小于则跳转
    JLE 有符号小于等于则跳转（同JNG）
    JNLE 有符号不小于等于则跳转（同JG）

修改.en_frame段权限

IDA修改

通过IDA打开程序，在IDA反汇编头部，也就是最开始的地方（一般是LOAD段）的起始开始找，如下，一段一段的找TYPE: .EN_FRAME的那段

找到这段后，我们可以使用工具对其进行修改内容，我用的是ida8.4存在自带工具，对于低版本的可以使用keypatch工具。

选中Flags对应的那行中的那个字节4，在菜单栏中点Edit->Patch program->change byte，将首字节中的04改成07，然后点击ok即可。意思是将该段的权限改成可读可写可执行。

对于需要执行跳转jmp到.en_frame段的patch都要对该段的权限进行检查，不然patch完可能也无法执行，这一步很关键。

上述方法可能有些时候并不管用，如果你觉得按上述修改了以后还是不能正常执行的话，那我们就采用下面的方法在进行执行权限的修改

可以知道在地址0x40123C位置的汇编指令是取出buffer变量的地址，放到rax中；地址0x401240位置的汇编指令是将地址放置到寄存器rdi中（第一个参数，也是唯一的参数），所以我们的jmp指令应该放置在0x401240~0x401243之间，因为要保证有5个字节的剩余给放jmp指令。

修改完以后的效果；

mov rdi, 0: read函数的第一个参数fd，从标准输入中读取字节

mov rsi, rax: read函数的第二个参数，读入的字节存放的位置

mov rdx, 80h: read函数允许读入字节的最大长度

mov eax, 0: syscall函数的系统调用号，这个设置为0，表示通过syscall调用read函数（如果是32位程序，就设置为3）

syscall: 执行syscall

jmp loc_40124D: 跳转回原来的命令执行顺序（call _gets 的下一条指令）

ciscn分区赛参赛感悟

首先介绍此次分区赛采用的赛制和题目分布，本次分区赛为两题web+一题pwn，赛制为awd；

1. 评判篇

第一，我作为一名pwn手进行参赛，自认为水平不佳，能看出漏洞点但是不会利用，相信这也是大多数pwn手存在的问题；所以本次分区赛由于awd赛制的性质，希望采用只修不打的战略去保住自己的分数（或者采用别的方式去打），但是最终得到的结论是由于权限锁死，无法替换文件，无法删除无法上传文件，简单来说，除了/tmp目录其他目录一律没有权限，不能修！只能打！我不清楚是不是由于技术等原因导致，但是我个人认为既然是awd赛制，基本权限和上传下载文件应该是基本的权限，我实在无法相信这就是你们口中说的，“前一天我们已经测试过了，没有问题”。本人对这里产生极大的不满！！！

第二，作为一名选手，比赛局域网的安全应该得到技术保障（awd还不是选手物理主机的攻防），为什么没有进行用户队伍之间的安全隔离？甚至允许队伍之间相互攻击导致赛前出现队伍之间植入木马的问题出现。

2. 检讨篇

第一，作为一名参赛选手，应该全方面考虑赛场的各种问题，包括不能修的问题，不应该将全部希望都寄托在保分上，这确实是本身考虑不周的问题。

第二，作为一只队伍，本身选取的修题保分战略确实存在相当多的问题，虽然不是作为队长，但是却对队伍的决策产生了比较大的影响，从而导致队伍的政策往该方向倾斜。

第三，作为一名技术人员，技术不佳才是最大的败笔，技术不佳者没有原因，都可以归咎于能力不足！继续加油吧！败者没有借口！
